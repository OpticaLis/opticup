## מצב קיים — Scenario 9: סריקה בכניסה

### איך זה עובד

ליד מגיע לאירוע ומציג QR Code. הנציג בכניסה סורק אותו בסורק ברקודים
ישירות לתוך בורד **Entrance - Scan QR** (Board 5088675161) ב-Monday.

ה-QR מכיל את ה-**Lead ID (Pulse ID) מ-Tier 3** — מזהה ייחודי לאירוע הזה בלבד.
בכל אירוע חדש הליד מקבל Lead ID חדש ב-Tier 3 (כי Tier 3 מתנקה בין אירועים).

**חיבור ל-Tier 2 (Master Board):** מתבצע דרך מספר טלפון — ב-Tier 2 יש הגבלת
ייחודיות על טלפון, אי אפשר להירשם עם אותו מספר גם אם השם שונה.

### פלואו

```
ליד מגיע לחנות → מציג QR Code
    │
    ├── נציג סורק QR בסורק ברקודים → הערך (Lead ID) נכנס לבורד Entrance
    │
    ├── Scenario 9 מופעל (webhook על שורה חדשה ב-Entrance)
    │   │
    │   ├── שליפת Lead ID מהסריקה
    │   │
    │   ├── חיפוש Lead ID ב-Tier 3 (Event Attendees)
    │   │   ├── נמצא + לא שומש → ✅ "המשתתף אומת בהצלחה!"
    │   │   ├── נמצא + כבר שומש → ❌ "המשתתף כבר ביצע הזמנה (הברקוד כבר...)"
    │   │   └── לא נמצא → ❌ "המשתתף לא רשום — נא לבדוק את הוואט..."
    │   │
    │   ├── עדכון בורד Entrance:
    │   │   ├── אישור כניסה: "אושר" (ירוק) / "שגיאה" (אדום)
    │   │   ├── שם המשתתף (נשלף מ-Tier 3)
    │   │   ├── Phone Number (נשלף מ-Tier 3)
    │   │   ├── הודעת מערכת (הסבר טקסטואלי)
    │   │   ├── צריך בדיקת ראייה (כן/לא — נשלף מ-Tier 3)
    │   │   └── סוג אירוע (SuperSale/MultiSale)
    │   │
    │   └── עדכון Tier 3: סטטוס → "הגיע"
    │
    └── ליד בלי QR (fallback ידני):
        ├── נציג מחפש את הליד בטיר 3 לפי טלפון או שם
        ├── מעתיק את ה-Lead ID הייחודי
        └── מדביק ידנית בבורד Entrance → אותו אפקט בדיוק
```

## מצב קיים — Scenario 10A: פעולות אחרי סריקה

### איך זה עובד

אחרי ש-Scenario 9 מאשר כניסה, Scenario 10A מעדכן את שאר הבורדים.

### פלואו

```
Scenario 9 מאשר כניסה (סטטוס "אושר")
    │
    ├── עדכון Tier 3 (Event Attendees):
    │   └── סטטוס → "הגיע לאירוע"
    │
    ├── עדכון Tier 2 (Master Board):
    │   ├── חיפוש לפי מספר טלפון (bridge בין Tier 3 ל-Tier 2)
    │   ├── סטטוס → "הגיע לאירוע"
    │   └── Notes += "הגיע לאירוע {{event_number}} | {{timestamp}}"
    │
    └── עדכון Events Management:
        └── Total Attended += 1 (דרך Monday Calculation Workflow)
```

### עמודת "פעולות סיום — אחרי תשלום"

**מצב קיים:** עדכון **ידני**. אחרי שהליד קונה בחנות, מישהו מעדכן את העמודה
בבורד Entrance ל-"טופס שביעות רצון" (ירוק). אם לא קנה — "לא קנה — לא לעשות
כלום" (אדום).

**מצב עתידי:** עדכון **אוטומטי** — כשמכירה נרשמת ב-Optic Up ERP (Module 1 —
Inventory/Sales), המערכת תזהה שהלקוח הוא משתתף באירוע ותעדכן אוטומטית.

## Scenario 10B: טופס שביעות רצון — סטטוס נוכחי

> **Scenario ID:** 8457806 | 12 מודולים | 30 ops
> **URL:** https://eu2.make.com/402680/scenarios/8457806/edit

### מצב קיים
שולח טופס שביעות רצון ללידים שעמודת "פעולות סיום" = "טופס שביעות רצון".
כרגע **לא בשימוש פעיל** — אין צורך מיידי.

### כיוון עתידי — CX Flow חכם

```
ליד קנה באירוע → מקבל טופס שביעות רצון (אוטומטי, 24 שעות אחרי)
    │
    ├── ביקורת חיובית (4-5 כוכבים):
    │   └── בקשה אוטומטית לביקורת בגוגל (עם קישור ישיר)
    │
    └── ביקורת שלילית/ניטרלית (1-3 כוכבים):
        └── התראה לנציג → שיחת follow-up אישית
```

## כיוון למערכת חדשה — Event Day Module

### מה משתנה

| נושא | מצב קיים | מערכת חדשה |
|------|----------|------------|
| סריקה | סורק ברקודים → Monday board | סורק / קאמרה → מסך כניסה ב-Optic Up |
| אימות | Make scenario מול Tier 3 | בדיקה ישירה ב-Supabase (Event Attendees table) |
| Fallback (בלי QR) | חיפוש ידני ב-Tier 3, העתקת ID | חיפוש לפי טלפון/שם במסך הכניסה, אישור בלחיצה |
| עדכון סטטוסים | 3 בורדים נפרדים (Entrance + Tier 3 + Tier 2) | טבלה אחת + view |
| "צריך בדיקת ראייה" | עמודה ב-Monday, הנציג רואה | מוצג בגדול במסך הכניסה אחרי סריקה |
| "פעולות סיום" | עדכון ידני אחרי רכישה | **אוטומטי** — חיבור ל-Module 1 (Sales) |
| טופס שביעות רצון | Scenario 10B, לא פעיל | Messaging Hub trigger — 24 שעות אחרי רכישה |
| CX routing | לא קיים | חיובי → גוגל, שלילי → שיחת נציג |

### נתונים שצריכים להיות בטבלת event_attendances (Supabase)

| שדה | סוג | תפקיד |
|-----|-----|--------|
| `id` | UUID | מזהה ייחודי |
| `tenant_id` | UUID | זיהוי שוכר (RLS) |
| `event_id` | UUID | FK לטבלת אירועים |
| `lead_id` | UUID | FK לטבלת לידים (Master) |
| `status` | TEXT | registered / confirmed / paid / arrived / purchased / cancelled |
| `registered_at` | TIMESTAMPTZ | מתי נרשם |
| `paid_at` | TIMESTAMPTZ | מתי שילם פיקדון (null = לא שילם) |
| `arrived_at` | TIMESTAMPTZ | מתי נכנס לאירוע (null = לא הגיע) |
| `purchased_at` | TIMESTAMPTZ | מתי קנה (null = לא קנה) — חיבור ל-Module 1 |
| `needs_eye_exam` | BOOLEAN | צריך בדיקת ראייה |
| `source` | TEXT | facebook / referral / qr / manual |
| `source_id` | TEXT | Lead ID של המפנה (אם referral) |
| `qr_code` | TEXT | ערך ה-QR הייחודי לאירוע זה |
| `notes` | TEXT | הערות |
