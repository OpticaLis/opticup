**Result: NO sequences exist in the `public` schema.**

This is a deliberate architectural choice, not a gap. All identifier generation in Optic Up uses one of these patterns instead of Postgres sequences:

1. **UUIDs for primary keys** — every table uses `gen_random_uuid()` or `uuid_generate_v4()` as the default for `id` columns. See `01-tables.md` for the table list and `02-columns.md` for the column-level defaults.

2. **Atomic RPCs for human-readable sequential numbers** — purchase order numbers, return numbers, box numbers, and internal document numbers are generated by SECURITY DEFINER functions in `05-functions.md`:
   - `next_po_number(p_tenant_id uuid, p_supplier_number text) → text`
   - `next_return_number(p_tenant_id uuid, p_supplier_number text) → text`
   - `next_box_number(p_tenant_id uuid) → text`
   - `next_internal_doc_number(p_tenant_id uuid, p_prefix text) → text`

   These functions are required by **Iron Rule #13** (sequential number generation must use atomic RPC with `FOR UPDATE` lock) to prevent race conditions in multi-tenant concurrent writes.

3. **Per-tenant counters in tenant_config or similar tables** — where applicable, counters are stored as rows in tables (with RLS isolation) rather than as global sequences, so each tenant has its own independent counter space.

**Reconciliation implication for `GLOBAL_SCHEMA.sql`:** the schema doc must NOT contain `CREATE SEQUENCE` statements. If any are present, they are stale and should be removed during Phase 3A Step 7 (GLOBAL_SCHEMA.sql rewrite).
