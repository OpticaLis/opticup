### Architecture Decision v1 (2026-04-21): Internal-First

Go-Live roadmap rewritten from C1вҖ“C9 (Make-centric) to P1вҖ“P7 (internal-first).

**Old approach (abandoned):** clone every Make scenario, replace Monday modules with HTTP calls to Supabase. Resulted in brittle Make scenarios (broken routers, `{{}}` conflicts, no error handling, missing business logic).

### Architecture Decision v2 (2026-04-22): Make = Send-Only Pipe

**Supersedes v1.** After P3b was built with native Supabase modules in Make (8 modules), Daniel decided Make should have **zero access to Supabase**. All DB operations move to a Supabase Edge Function.

**New architecture (final):**
- **Supabase Edge Function `send-message`** вҖ” fetches template, substitutes variables, writes log, then calls Make with a **ready-to-send message** (channel + recipient + subject + body). All logic is here.
- **Make = send-only pipe** вҖ” 3 modules only: Webhook вҶ’ Router вҶ’ SMS | Email. No Supabase modules, no DB connection, no variable substitution. Receives a fully prepared message and forwards it.
- **WhatsApp deferred** вҖ” Daniel plans to switch from Green API to Meta's official WhatsApp API. Third route added to Make when ready.
- **Template design** вҖ” each template has up to 3 channel variants (SMS/Email/WhatsApp), not all required. Email body is full HTML stored in DB. Variable format: `%name%` (not `{{}}`).
- **Ad-hoc broadcast** вҖ” Broadcast Wizard (B5) wired to send via same Edge Function in raw-content mode (no template needed).

**Old P3b scenario (9104395, 8 modules with Supabase)** вҶ’ replaced by P3c+P4 (3-module scenario + Edge Function + trigger wiring).

See `modules/Module 4 - CRM/go-live/ROADMAP.md` for full P1вҖ“P7 plan.
