# Automation Mockup A — Wizard Step-by-Step

## איך הוא בנוי

- **רשימת חוקים** (טבלה רגילה בעמוד הראשי) עם עמודת "בורד" צבעונית — לחיצה על "+ חוק חדש" או על כל שורה לעריכה פותחת **wizard מודאלי**.
- **ה-wizard ב-4 שלבים**:
  1. **בורד** — 4 כרטיסים גדולים: 📥 לידים נכנסים / 👥 רשומים / 📅 אירועים / ✅ נרשמים לאירוע. בחירה אחת חובה.
  2. **תנאי טריגר** — שדות מתאימים לבורד שנבחר:
     - לידים נכנסים: "תמיד" / "מקור הליד שווה ל-X".
     - רשומים: "סטטוס משתנה ל-Y".
     - אירועים: "סטטוס אירוע משתנה ל-Z" / "ספירה עוברת סף".
     - נרשמים לאירוע: "סטטוס הרשמה = registered" / "= waiting_list".
  3. **נמענים** — אפשרויות מסוננות לבורד: "הליד עצמו" / "כל Tier 2" / "Tier 2 חוץ מנרשמים" / "נרשמים לאירוע" / "רשימת המתנה". + סינון לפי סטטוס ליד אם רלוונטי.
  4. **תבנית + ערוצים + סיכום** — dropdown תבניות מסונן לבורד; checkboxes ערוצים; משפט סיכום בעברית.
- **Step indicator** עליון מראה ✓ לשלבים שהושלמו.
- **Footer** עם "חזרה / המשך / שמור" — לא ניתן להמשיך אם השלב הנוכחי לא תקין.

## יתרונות

1. **בלתי אפשרי לטעות**: כל שלב מוודא מילוי לפני המעבר. אי-אפשר לשמור חוק ללא תבנית, ללא ערוצים, וכו'.
2. **חינוכי**: משתמש חדש (גם Daniel ביצירה ראשונה) לומד מבנה החוק תוך כדי המעבר.
3. **שדות מתאימים לבורד** — מקטין את הרעש (דניאל לא רואה "מקור שווה ל-" באירועים, כי זה לא רלוונטי שם).
4. **משפט סיכום בעברית בשלב 4**: "ברגע שמישהו ישנה סטטוס אירוע ל-X, יישלח SMS+EMAIL ל-Y..." — בדיקת הגיון לפני שמירה.

## חולשות

1. **קליקים רבים לעריכה מהירה**: שינוי תבנית בחוק קיים = פתיחת wizard, מעבר לשלב 4. Step indicator עליון מאפשר קפיצה אבל זה עדיין 2 קליקים.
2. **בלתי-נראה את החוק כולו במבט אחד** במהלך העריכה — תמיד שלב אחד מוצג בלבד.
3. **גודל**: 4 שלבים מאריכים את הזמן ליצירת חוק ראשון לעומת form יחיד.
4. **State management מסובך**: שדות שלב 2 תלויים בבחירת שלב 1. אם משתמש חוזר ושינה בורד, צריך לאפס/לשמר את שלב 2?

## איך הוא מטפל ב-4 קטגוריות הטריגר של Daniel

**מצוין — הקטגוריות הן המבנה הראשי של ה-wizard:**

| קטגוריה Daniel | בורד ב-wizard | טריגר אפשרי בשלב 2 | נמענים אפשריים בשלב 3 |
|---|---|---|---|
| A. שינוי סטטוס אירוע | 📅 אירועים | סטטוס אירוע משתנה / ספירה עוברת סף | Tier 2 / Tier 2 ללא נרשמים / attendees / waiting_list |
| B. שינוי סטטוס ליד | 👥 רשומים | סטטוס ליד משתנה | הליד עצמו |
| C. הרשמה למערכת (lead_intake) | 📥 לידים נכנסים | תמיד / מקור שווה ל- | הליד עצמו |
| D. הרשמה לאירוע + סטטוס | ✅ נרשמים לאירוע | סטטוס הרשמה = registered / waiting_list | הליד עצמו |

כל 13 החוקים הקיימים על demo נופלים בדיוק לאחת מ-4 הקטגוריות. אין צורך בקטגוריה חמישית.

## כמה זמן ייקח לבנות

**~16–22 שעות עבודה.**

| Sub-task | שעות |
|---|---|
| State machine (4 שלבים, validation per step, navigation) | 3–4 |
| שלב 1: 4 כרטיסי בורד (UI) | 1 |
| שלב 2: 4 וריאנטים של שדות תנאי לפי בורד + reveal logic | 3–4 |
| שלב 3: 4 וריאנטים של recipient picker + sub-filter (סטטוס ליד) | 2–3 |
| שלב 4: dropdown תבניות מסונן + checkboxes ערוצים + summary sentence builder | 2–3 |
| חזרה לאחור: שמירת state בין שלבים, אזהרה אם משנים בורד שמשנה את כל המבנה | 1–2 |
| פתיחת wizard למצב עריכה (קפיצה ישירה לשלב 4 לעריכה מהירה?) | 1–2 |
| QA — כל שילוב טריגר/תנאי/נמען על demo + verify על rules קיימים | 2 |
| RTL polish | 1 |
