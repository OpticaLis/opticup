# Automation Mockup C — Single Form (Conditional Fields)

## איך הוא בנוי

- **Filter bar** בראש הרשימה הראשית: pills של 4 בורדים (📥 / 👥 / 📅 / ✅) + ספירת חוקים בכל אחד.
- **כפתור "+ חוק חדש"** פותח modal/page עריכה.
- **טופס יחיד** עם זרימה מלמעלה למטה:
  1. **שם החוק** (input).
  2. **בחירת בורד** — 4 כרטיסים צבעוניים גדולים. בחירה חובה.
  3. **בלוק תנאי** — בצבע הבורד, נחשף **רק לאחר בחירה**:
     - ב-`אירועים`: dropdown סטטוס + dropdown נמענים + סינון סטטוס ליד.
     - ב-`רשומים`: dropdown סטטוס ליד + נמענים = הליד עצמו (קבוע).
     - ב-`לידים נכנסים`: radio (תמיד / מקור) + נמענים = הליד עצמו (קבוע).
     - ב-`נרשמים לאירוע`: dropdown סטטוס הרשמה + נמענים = הליד עצמו (קבוע).
  4. **תבנית הודעה** (dropdown מסונן לבורד).
  5. **ערוצים** (3 checkbox cards צבעוניים).
  6. **בלוק "סיכום בעברית"** — משפט טבעי שמתעדכן לפי הבחירות.
- **Footer**: ביטול / שמור.

## יתרונות

1. **הכי קל לבנות.** המבנה של ה-modal הקיים נשאר; רק ה-rendering של השדות מותנה בבורד.
2. **הכל במסך אחד**. אין wizard, אין גלילה רחבה — Daniel רואה את כל החוק בבת אחת ויכול לחזור על שדה.
3. **בלוק "סיכום בעברית"** — Daniel קורא משפט אנושי שלם ומבין מיד מה החוק עושה. תיקון חיוני לפי הביקורת ("כמעט בלתי אפשרי לבנות אוטומציות לבד").
4. **Filter bar בראש** = ניווט מהיר לבורד הרצוי, וגם קונטקסט של "כמה חוקים יש בכל בורד".
5. **דניאל לא רואה שדות לא רלוונטיים**: "מקור שווה ל-" נעלם בבחירת אירועים. "ספירה עוברת סף" מוצג רק בקונטקסט שמתאים לו.

## חולשות

1. **טופס ארוך** במצב מלא (אחרי בחירת בורד) — גלילה אנכית סבירה אבל לא קצרה.
2. **פחות "מודרך"** מ-wizard: אם דניאל לא ממלא שדה חובה, הוא יקבל error בעת שמירה ולא prevention בזמן עריכה.
3. **שינוי בורד באמצע עריכה = איפוס בלוק התנאי**. צריך אזהרה ("שינוי בורד יאפס את התנאים, להמשיך?") כי בלוק התנאי משתנה במלואו.
4. **הכי קרוב ל-UI הקיים** — אם הביקורת היא "ה-UX לא טוב מספיק", השיפור פחות דרמטי מ-wizard או flowchart.

## איך הוא מטפל ב-4 קטגוריות הטריגר של Daniel

**הקטגוריות הן בחירת הבורד הראשונה — מבנית זהה ל-Mockup A.**

| קטגוריה Daniel | בורד שיוצג | בלוק תנאי שנחשף |
|---|---|---|
| A. שינוי סטטוס אירוע | 📅 אירועים | סטטוס אירוע + נמענים + סינון ליד |
| B. שינוי סטטוס ליד | 👥 רשומים | סטטוס ליד |
| C. lead_intake | 📥 לידים נכנסים | תמיד / מקור |
| D. הרשמה לאירוע | ✅ נרשמים לאירוע | סטטוס הרשמה |

בכל הבורדים, בלוק התבנית + ערוצים + סיכום מופיעים אחרי. רחוק יותר מ-state החוקים הקיים, אבל מוכר ל-Daniel.

## כמה זמן ייקח לבנות

**~10–14 שעות עבודה.** הכי קל. רוב הקוד הקיים נשמר.

| Sub-task | שעות |
|---|---|
| Filter bar + ספירת חוקים פר בורד (משתמש בשאילתה GROUP BY trigger_entity) | 1–2 |
| 4-כרטיס בורד picker בראש הטופס + state | 1 |
| Conditional rendering של בלוק התנאי לפי הבורד הנבחר (4 וריאנטים) | 3–4 |
| Refactor של ה-existing fields ל-board-aware (recipients dropdown מסונן, status filter contextual) | 2 |
| Template dropdown מסונן לבורד (JOIN בין `crm_message_templates.slug` בסיס לבורדי החוקים שמשתמשים בהם) | 1 |
| בלוק "סיכום בעברית" — sentence builder לפי בחירות, 4 templates של משפט (אחד לכל בורד) | 2 |
| Edit mode: פתיחת חוק קיים → קוד בודק את trigger_entity ו-trigger_event → פותח את הבורד הנכון אוטומטית | 1 |
| Validation: שדות חובה לפי בורד, אזהרה לשינוי בורד | 1 |
| QA + RTL polish | 1–2 |
