## Summary

Added a "ОАОъОЪ ОЏОљОЎОЕОе ОфОДОаОЋОЪ" button to the lead detail modal's Details tab so
staff can manually approve `terms_approved=true` on a tier-1 lead that came

The button is gated to `lead.terms_approved === false` AND `tier === 1`

`UPDATE crm_leads SET terms_approved=true, terms_approved_at=now()` Рєњ
`crm.lead.terms_approved_manual` activity log Рєњ toast Рєњ modal refresh.

After approval, the existing status-dropdown / transfer-to-tier2 path on

## Decisions made in real time

1. **Where to place the button** Рђћ Daniel said "next to the ОфОаОљОЎОЮ row".
   The Details tab's `row('ОфОаОљОЎОЮ', ...)` already had an `extra` slot
   (used by the resubscribe button on the ОЕОЎОЋОЋОД row). Reusing the same
   pattern means zero new layout primitives.
2. **Modal.confirm vs window.confirm** Рђћ chose `Modal.confirm` per
   project convention (auto-memory feedback). Hebrew text, emerald
   confirm button to match the success semantic.
3. **Where to put the new helpers** Рђћ `crm-leads-detail.js` was at 349
   lines, hitting Rule 12's hard cap of 350. Inlining the wire/render
   helpers there would have pushed it to 368. Moved them into
   `crm-lead-actions.js` (was 284 Рєњ 342, soft warning only). This keeps
   the actions module as the single home for lead mutation logic.
4. **Tier gating** Рђћ `termsApproveButtonHtml` returns empty for
   `tier !== 1`. Reasoning: a tier-2 lead is already past the
   terms-approval gate; if it somehow has `terms_approved=false`, that
   is a data anomaly that warrants its own SPEC, not a one-click flip.
5. **Refresh strategy** Рђћ instead of reloading the full modal, the
   `refresh` closure re-renders just the details tab body. After
   approval the button is regenerated by `termsApproveButtonHtml`,
   which now returns empty (because `lead.terms_approved` was mutated
   in-memory before the refresh). The ОфОаОљОЎОЮ row flips to
   'РюЁ ОљОЋОЕОеОЋ' on the same click, no modal flicker.
6. **Parent-list refresh** Рђћ also called `reloadCrmIncomingTab` and
   `reloadCrmLeadsTab` if defined. That way the badge/icon on the
   incoming tab updates without a manual refresh.
